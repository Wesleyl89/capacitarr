# Insights Page Redesign

**Created:** 2026-03-19T19:15Z
**Status:** 📋 Planned

## Summary

Complete redesign of the Insights page to focus exclusively on capacity management. Replace boring/irrelevant charts with visually striking, capacity-focused visualizations using ECharts' advanced chart types. Fix backend data issues (show/season duplication, bloat comparison across types, unfiltered protected content). Redesign all 6 theme color palettes for chart aesthetics.

## Context

The original Insights page had charts that were either broken (oklch color parsing — fixed in prior commit on `fix/chart-color-parsing`), irrelevant to capacity (genre distribution, request fulfillment), duplicative (quality donut vs quality bar), or had bad data (size anomalies comparing shows to episodes, ranked lists showing both shows and their seasons). The page used tabs that fragmented the information.

## Phase 1: Theme Color Palette Redesign

### Problem Analysis

The current palettes were generated by taking the theme's primary oklch color and creating 3 variations by shifting hue ±15-25° and adjusting lightness. This produces palettes that are too similar — all colors in the same narrow hue range.

Additionally, the `generatePalette()` function in `useEChartsDefaults.ts` spreads hues across a 120° arc from chart1, which for violet (hue 270°) produces purple → magenta → pink → salmon. This is the "purple towards pink" problem.

### Current Palettes (problems noted)

| Theme | chart1 | chart2 | chart3 | chart4 | Problem |
|-------|--------|--------|--------|--------|---------|
| **Violet** | `#8e51ff` (purple) | `#6054ec` (blue-purple) | `#a96de6` (magenta) | `#c77dd8` (pink) | Skews warm/pink. No cool contrast. |
| **Ocean** | `#0080ce` (blue) | `#0077a2` (teal-blue) | `#0086d8` (blue) | `#00a7b1` (teal) | All too similar — 4 shades of blue-teal. No contrast between slices. |
| **Emerald** | `#00a056` (green) | `#098926` (dark green) | `#00ab8a` (teal) | `#83ae53` (olive) | chart2 too dark, chart4 muddy olive. |
| **Sunset** | `#f07900` (orange) | `#dd6836` (red-orange) | `#e79b3d` (gold) | `#dd4115` (red) | chart4 is aggressive red, clashes with destructive color. |
| **Rose** | `#d72f92` (magenta) | `#b4319b` (purple) | `#e45580` (pink) | `#f16f7e` (salmon) | Same pink-skew problem as violet. |
| **Slate** | `#4c5666` (gray) | `#3e4955` (dark gray) | `#6c7181` (medium gray) | `#77828c` (light gray) | Intentionally muted — OK for monochrome theme. |

### Redesigned Palettes

Design principles:
1. **Complementary spread**: chart colors should span at least 90° of hue wheel for visual distinction
2. **Consistent lightness**: all 4 colors should have similar perceived brightness (no dark outliers)
3. **Cool bias for warm themes**: warm themes (violet, sunset, rose) get cooler accent colors for contrast
4. **Warm bias for cool themes**: cool themes (ocean, emerald) get warmer accent colors
5. **chart1 = theme primary**: always matches the theme identity
6. **chart2-4 = complementary accents**: visually distinct from chart1 and from each other

### Step 1.2: Update `generatePalette()` in `useEChartsDefaults.ts`

The current function spreads hues across a 120° arc from the base color. For charts with many slices (like the sunburst), this still produces too-similar colors.

Change the algorithm to use a **golden angle** (137.5°) hue rotation, which produces maximally distinct colors regardless of count:

```typescript
function generatePalette(baseHex: string, count: number): string[] {
  const hsl = hexToHSL(baseHex);
  const goldenAngle = 137.508;
  return Array.from({ length: count }, (_, i) => {
    const hue = (hsl.h + i * goldenAngle) % 360;
    const lightness = isDark.value ? 55 + (i % 3) * 8 : 45 + (i % 3) * 8;
    return hslToHex(hue, Math.max(hsl.s, 50), lightness);
  });
}
```

## Phase 2: Backend Data Fixes

### Step 2.1: Fix size anomalies — group by `(qualityProfile, type)`

**File:** `backend/internal/services/analytics.go`

Currently `GetSizeAnomalies()` groups all items by quality profile only. A show (380 GB) is compared against the median of all items in that profile, including seasons (14 GB) and movies (4 GB).

Fix: change the grouping key from `qualityProfile` to `(qualityProfile, mediaType)`. The median for "HD-1080p shows" is compared only against other "HD-1080p shows", and "HD-1080p movies" against other "HD-1080p movies".

Add a `MediaType` field to the `SizeAnomaly` struct so the frontend can display it.

### Step 2.2: Exclude protected items from dead/stale content and size anomalies

**Files:** `backend/internal/services/watch_analytics.go`, `backend/internal/services/analytics.go`

The `WatchAnalyticsService` and `AnalyticsService` need access to the rules engine to check protection status. Add a `RulesSource` interface dependency:

```go
type RulesSource interface {
    GetEnabledRules() []db.CustomRule
}
```

In `GetDeadContent()`, `GetStaleContent()`, and `GetSizeAnomalies()`, after filtering for enrichment data, evaluate each item against the active rules using `engine.ApplyRules()`. Exclude items where the result is `always_keep` (absolute protection). Items with `prefer_keep` or `lean_keep` should still appear — they're soft preferences, not absolute protections.

Update the response structs to include a `protectedCount` field so the UI can show "X items excluded (protected by rules)".

### Step 2.3: Deduplicate ranked lists — exclude season items

**File:** `backend/internal/services/watch_analytics.go`

In `GetPopularity()`, filter `enrichedItems` to exclude `type == "season"` before building the top/bottom ranked lists. Shows aggregate their seasons' play counts, so including both double-counts.

### Step 2.4: Add capacity forecast endpoint

**File:** `backend/internal/services/metrics.go` (new method)

Add `GetCapacityForecast()` that:
1. Fetches the last 30 days of metrics history
2. Performs linear regression on `usedCapacity` over time
3. Returns: current usage %, growth rate (GB/day), projected days until threshold, projected days until 100%

```go
type CapacityForecast struct {
    CurrentUsedPct    float64 `json:"currentUsedPct"`
    GrowthRatePerDay  int64   `json:"growthRatePerDay"`  // bytes/day
    DaysUntilThreshold int    `json:"daysUntilThreshold"` // -1 if shrinking
    DaysUntilFull     int     `json:"daysUntilFull"`      // -1 if shrinking
    TotalCapacity     int64   `json:"totalCapacity"`
    UsedCapacity      int64   `json:"usedCapacity"`
}
```

**File:** `backend/routes/analytics.go` — add `GET /api/v1/analytics/forecast`

### Step 2.5: Add storage-by-type-and-quality endpoint

**File:** `backend/internal/services/analytics.go` (new method)

Add `GetStorageSunburst()` that returns hierarchical data for the sunburst chart:

```go
type SunburstNode struct {
    Name     string         `json:"name"`
    Value    int64          `json:"value"`    // bytes
    Children []SunburstNode `json:"children,omitempty"`
}
```

Structure: root → [movies, shows, seasons] → [quality profiles within each type]

**File:** `backend/routes/analytics.go` — add `GET /api/v1/analytics/storage-breakdown`
